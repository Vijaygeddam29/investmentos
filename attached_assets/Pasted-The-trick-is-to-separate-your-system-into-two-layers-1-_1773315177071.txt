The trick is to separate your system into two layers:

1️⃣ Heavy data pipeline (runs in batches)
2️⃣ Fast query layer (the dashboard)

Most people try to make the dashboard calculate everything live, which makes the system slow or crash when the universe gets large. Instead, you pre-compute everything and store it, then the UI only reads results.

This is how large quant systems operate.

🧠 The Trick: Pre-Computed Factor Warehouse

Instead of doing this:

Dashboard request
   ↓
Call FMP API
   ↓
Compute factors
   ↓
Return result

You do this:

Nightly pipeline
   ↓
Fetch data (FMP)
   ↓
Compute 120 factors
   ↓
Store results in database
   ↓
Dashboard reads precomputed scores

The UI becomes instant, even with thousands of stocks.

How Big Firms Do It

Large quant shops run factor pipelines overnight and store the results in a factor database.

Examples:

Bridgewater Associates

Two Sigma

Renaissance Technologies

Their dashboards never compute live factors.

They only query precomputed tables.

What Your Current System Likely Does

Right now your system probably looks like this:

Pipeline → scores → dashboard

But it runs every time you press “Run Pipeline”.

That’s fine for 5–50 stocks, but not for 2000+ stocks.

The Upgrade

Add a Factor Warehouse table.

Example schema:

factor_snapshots

ticker
date
fortress_score
rocket_score
wave_score
entry_score

profitability_score
growth_score
efficiency_score
cashflow_score
valuation_score

rsi
macd
momentum
margin_of_safety

Now every pipeline run stores a daily snapshot.

Your charts and screeners read from this table.

Why This Scales to 10,000 Stocks

Because:

heavy compute = once per day
queries = simple database reads

Example query:

SELECT *
FROM factor_snapshots
WHERE fortress_score > 0.8
AND market_cap > 10B

That takes milliseconds.

The Second Trick (Even Better)

Use incremental updates.

Instead of recalculating everything:

Update only stocks whose data changed

Example triggers:

earnings release
price movement >5%
new financial statements
